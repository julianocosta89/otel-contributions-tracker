# data/

This directory contains JSON files generated by the pipeline scripts and committed to the repository. The web app (`index.html`) loads these files at startup to avoid live network requests for the default view.

## Files

| File | Generated by | Refresh frequency |
|------|-------------|-------------------|
| `cache.json` | `scripts/fetch-data.mjs` | Daily (GitHub Actions) |
| `sigs.json` | `scripts/fetch-sigs.mjs` | Daily (GitHub Actions) |
| `affiliations.json` | `scripts/fetch-affiliations.mjs` | Weekly (GitHub Actions) |
| `github-companies.json` | `scripts/fetch-github-companies.mjs` | Weekly (GitHub Actions, requires `GITHUB_TOKEN`) |
| `roles.json` | `scripts/fetch-roles.mjs` | Weekly (GitHub Actions) |

---

### cache.json

Main data cache. Used directly by the app when the user has not changed any filter from the defaults (1-year window, all platforms, all activity types).

```
{
  "fetchedAt": "<ISO timestamp>",
  "sources": {
    "periods": {
      "<preset>": {
        "fetchedAt": "<ISO timestamp or date>",
        "status": "fresh | cached | cached-fallback"
      }
    },
    "filterCombos": {
      "<preset>": {
        "<platform>": {
          "fetchedAt": "<ISO timestamp or date>",
          "status": "fresh | cached | cached-fallback"
        }
      }
    }
  },
  "periods": {
    "<preset>": { ... }   // presets: 30d, 90d, 6m, 1y, 2y, 3y, all
  },
  "filterCombos": {
    "<preset>": {
      "<platform>": { ... }  // platforms: github, git, gerrit, gitlab, confluence, jira
    }
  }
}
```

**`periods[preset]`** — full leaderboard data for a time window:

| Field | Description |
|-------|-------------|
| `period` | `{ startDate, endDate }` strings |
| `activeContributors` | Summary counts with period-over-period delta |
| `activeOrganizations` | Summary counts with period-over-period delta |
| `contributorDependency` | Top-N contributor concentration stats |
| `organizationDependency` | Top-N organization concentration stats |
| `geographicalDistribution` | Country-level contributor breakdown |
| `contributors` | `{ total, data[] }` — full ranked contributor leaderboard |
| `organizations` | `{ total, data[] }` — full ranked organization leaderboard |

Each contributor record in `data[]` contains: `avatar`, `name`, `contributions`, `percentage`, `roles`, `githubHandleArray`, `previousContributions`.

Each organization record contains: `logo`, `name`, `contributions`, `percentage`, `website`, `previousContributions`.

`previousContributions` enables period-over-period comparison in the UI (e.g., rank change, contribution delta).

**`filterCombos[preset][platform]`** — same shape as a `periods[preset]` entry, but scoped to a single platform. Populated for all preset × platform combinations so the UI can switch platforms without making live API calls.

**`sources`** — per-period and per-platform freshness metadata. `fetchedAt` at the top of the file records the script run time; `sources` records the freshness of each cached slice. A `cached-fallback` status means a refresh failed for that slice and the script reused an older cached copy instead of silently replacing it with empty or partial data.

### sigs.json

Per-repository contributor and organization leaderboards. Used by the SIG/repo breakdown view.

```
{
  "fetchedAt": "<ISO timestamp>",
  "sources": {
    "repos": {
      "fetchedAt": "<ISO timestamp>",
      "status": "fresh"
    },
    "periods": {
      "<preset>": {
        "fetchedAt": "<ISO timestamp>",
        "status": "fresh | partial"
      }
    }
  },
  "repos": ["<repo-name>", ...],
  "periods": {
    "<preset>": {
      "<repo-name>": {
        "contributors": { "total": N, "data": [...] },
        "organizations": { "total": N, "data": [...] }
      }
    }
  }
}
```

`repos` is the list of all non-archived repositories in the `open-telemetry` GitHub organization. Presets match those in `cache.json`: `30d`, `90d`, `6m`, `1y`, `2y`, `3y`, `all`.

When a SIG period has `status: "partial"`, the period was refreshed but one or more repositories reused cached fallback data. The source entry may include `fallbackRepos`, `fallbackFrom`, and `errors` for troubleshooting.

### affiliations.json

Maps GitHub handles (lowercase) to current employer. Sourced from [CNCF gitdm](https://github.com/cncf/gitdm). This is the **highest-confidence** affiliation source.

```json
{
  "handle": {
    "company": "Acme Corp",
    "file": 3,
    "line": 42
  }
}
```

`file` and `line` are the source file index and line number within the gitdm dataset, useful for tracing the origin of an entry.

---

### github-companies.json

Maps GitHub handles (lowercase) to the company string from the user's GitHub profile. Used as a fallback when a handle is not present in `affiliations.json`. Values may be `null` when the handle was fetched but the user has no company set.

```json
{
  "handle": "Acme Corp",
  "other-handle": null
}
```

---

### roles.json

Maps GitHub handles (lowercase) to their highest GitHub team role within the `open-telemetry` org. Used to annotate contributors in the leaderboard.

```
{
  "fetchedAt": "<ISO timestamp>",
  "roles": {
    "handle": {
      "role": "maintainer | approver | code-owner | triager",
      "teams": ["team-slug", ...]
    }
  }
}
```

Role hierarchy (highest wins): `maintainer` > `approver` > `code-owner` > `triager`.

---

## Affiliation resolution

When the app resolves a contributor's employer it checks sources in this order, stopping at the first match:

1. `affiliations.json` — CNCF gitdm (highest confidence)
2. `github-companies.json` — GitHub profile company field (fallback)

If neither source has the contributor, the app shows no company affiliation (the LF Insights API response does not include an organization field used by the app).

## How the app uses these files

- On load, `cache.json` is fetched and used to populate the default view (1-year window, all platforms, all activity types).
- When the user selects a specific platform, the app reads from `cache.json`'s `filterCombos` section instead of making a live API call.
- When the user selects a non-default filter that is not covered by `filterCombos` (e.g., a custom date range or activity type), the app falls back to live LF Insights API requests.
- `sigs.json` is loaded on demand when the user opens the per-repository breakdown.
- `affiliations.json`, `github-companies.json`, and `roles.json` are loaded once and held in memory to annotate contributors throughout the session.
